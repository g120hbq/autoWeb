58微信小程序的现状与未来

微信小程序介绍

  • 微信内嵌web view 控件，本质上就是一个定制化的web浏览器
  • js-sdk类似vue和ng的mvvm思想编写
  • js里无法接触到dom,jQuery是完全不可以使用的

  • 官方开发工具功能简陋

  • 完全封闭的环境不等于浏览器环境

  • WXML/WXSS/JS格式
  • WXML 不等于 HTML
  • WXSS 不等于 CSS3
  • JS：支持ES6，ES5，不支持DOM
  • 不支持WEBGL,包含部分Canvas API
  • 封闭的发布渠道：程序需通过官方IDE提交审核
  • 腾讯完全控制客户端底层的内部实现
 
## 小程序的优势：    

1. 不用安装，即开即用，用完就走。省流量，省安装时间  
2. 体验上虽然没法完全媲美原生APP，但综合考虑还是更优  
3. 对于小程序拥有者来说，开发成本更低，他们可以更多财力，人力，精力放在如何运营好产品，做好内容本身  
4. 对于用户来说，相较于各种APP，微信小程序UI和操作流程会更统一  
这也会降低用户的使用难度  
5. 对于小程序拥有者来说，相较于原生APP，推广更容易更简单，更省成本

开发中发现的难点或者不好的地方

  1. 无法调用npm包
  2. 无法使用babel转码
  3. 无法重用组件(像React那样重用组件功能）
  4. 异步请求5条并发上限,否则报错

  5. 最多5个页面队列,否则不再跳转

    • wx.navigateTo
    • wx.redirectTo
    • wx.navigateBack

  6. wx.request 不是promise化的接口

项目中发现的问题

  1. 公共功能抽离不够
  2. 命名规范不够统一
  3. 自动构建功能不够友好

未来技术转型-两种微信小程序组件化开发方案

  1. 方案1 wepy 来自腾迅团队的解决方案,类vue
  2. 方案2 labrador

未来技术转型-项目代码设计层面

  1. 考虑性能--debounce throttle 函数去抖 函数节流 更简单的抽象化处理
  2. 考虑维护性--将常量 与 数据请求部分抽离出来
  3. 将部分常用WXAPI 封装一层，简化或降低BUG率

总结

  1. 组件化开发对项目开发效率质的提升
  2. 小程序还在初始发展,后续会越来越跟上前端的步伐
  3. 好的初始设计规划,能避免在项目大型化后遇到的各种尴尬
  4. 微信公众号最终会替换成微信小程序
